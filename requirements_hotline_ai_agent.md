## 第1章　概要・設計思想

### 1-1. プロトタイプの目的

店舗で発生したトラブル（調剤ミス・個人情報漏洩・管理薬剤の紛失・クレーム・事故・法令違反等）について、店舗スタッフがチャット形式でAIエージェントと対話することで、必要な情報を漏れなく構造化してホットラインへ伝達する仕組みを実装する。

### 1-2. 設計の前提

Why分析の過程で確認された制約条件を、設計の前提として組み込んでいる。

- **全件電話ルールは維持：** 店舗判断による見逃しリスクを排除する。AIエージェントは店舗の判断を肩代わりしない受付チャネル
- **店舗スタッフに書かせない：** 選択式を最大化。やむを得ない自由入力はAI要約で整理
- **薬剤師の専門判断は人が行う：** ミスレベル（Ⅰ〜Ⅴ）の判定はAIではなくホットライン担当が後から実施
- **AIは判定しない：** エスカレーション要否の判定は廃止。すべての報告をSlackへ通知し、人間が判断する

### 1-3. 設計の7原則

| No. | 原則 | 内容 |
|:---:|---|---|
| ① | **AIは判定しない** | すべての報告をSlackに通知し、判断は人間が行う。AIの誤判定リスクを構造的にゼロに |
| ② | **店舗スタッフに書かせない** | 選択式を最大化、自由入力はAI要約で整理。文章力の個人差を吸収 |
| ③ | **専門判断はHL担当が行う** | ミスレベル判定など薬剤師知見が必要な項目は店舗スタッフに聞かない |
| ④ | **記憶の流れに沿った質問順序** | 「いつ・どこで → 誰に → 何を → どうなった → どうしている」の自然な順序 |
| ⑤ | **回答に応じて質問を動的調整** | 服用なしなら回数を聞かない、管理薬剤紛失なら患者状態を聞かないなど |
| ⑥ | **任意のタイミングで修正可能** | 会話途中・確認画面の双方で項目単位の修正が可能 |
| ⑦ | **電話のフォールバックを残す** | 不安な場合はいつでもホットラインに電話できる選択肢を保持 |

## 第2章　対話フローの全体像

### 2-1. フェーズ構成（10ステップ）

AIエージェントとの対話は以下の10ステップで構成される。一部は条件分岐により自動的に挿入・スキップされる。

| Step | フェーズ | 内容 |
|:---:|---|---|
| **00** | 大分類選択 | 4つのカテゴリから選択（Ⅰ.調剤ミス／Ⅱ.個人情報漏洩／Ⅲ.管理薬剤の紛失／Ⅳ.その他トラブル） |
| **01** | ミス種類選択 | （調剤ミスのみ）14種のサブ分類から選択 |
| **02** | 基本情報 | 店番（6桁・必須バリデーション）／店名／医療機関名（調剤ミスのみ）等 |
| **03** | 事象の詳細 | カテゴリ・サブ分類に応じた質問。正誤情報・漏洩内容・紛失薬剤等 |
| **04** | 影響の確認 | （調剤ミスのみ）服用の有無 → 服用回数 → 健康被害 → 症状詳細 |
| **05** | 状況の確認 | 患者様の反応（管理薬剤紛失を除く）／発覚経緯 |
| **06** | 対応状況 | 未対応／対応中／対応予定あり／対応済み → 未対応以外なら対応内容を追加 |
| **07** | 補足事項 | 他に伝えておきたいこと（任意入力） |
| **08** | 入力内容の確認 | 全項目を一覧表示。項目単位で修正可能 |
| **09** | Slack通知＋管理表記入 | #ホットライン報告に自動投稿、PSV等に自動メンション、管理表に自動記入 |

### 2-2. 質問順序の設計思想

報告者の記憶の流れに沿った質問順序を採用。事実情報を先に固めてから判断や解釈が必要な情報に進むことで、認知負荷を最小化する。

| 段階 | 質問項目 |
|---|---|
| **いつ・どこで** | 店舗 → 医療機関 → 投薬日 → 発覚日 |
| **誰に** | 新患/既患 → 年齢層 |
| **何を** | ミス種類 → 正誤情報 |
| **どうなった** | 服用有無 → 服用回数 → 健康被害 → 症状詳細 |
| **どうしている** | 患者状態 → 発覚経緯 → 対応状況 → 対応内容 |

### 2-3. 重要な順序ルール

> **事象を対応状況より先に確認する**
>
> 調剤ミス以外のカテゴリで、「事象の詳細」を「対応状況」より先に聞く順序を採用。「何が起きたか」を完全に把握した後に「どう対応しているか」を聞くことで、情報の因果関係が明確になる。

| Ⅴ-2 | **健康被害の有無** | 選択＋テキスト | あり（内容を記載）/ なし ※「あり」の場合のみⅤ-3へ |

## 第4章　動的分岐ロジック

### 4-1. 動的分岐の主要パターン

AIエージェントは前問の回答内容に応じて、次に聞く質問を動的に挿入・スキップする。不要な質問を省くことで認知負荷を最小化する。

| No. | トリガー（条件） | 動作 |
|:---:|---|---|
| ① | 大分類「Ⅰ.調剤ミス」を選択 | → ミス種類（14種）の質問を挿入 |
| ② | 服用の有無で「服用した」を選択 | → 服用回数（何回／何日分）の質問を挿入 |
| ③ | 健康被害で「あり」を選択 | → 症状詳細の質問を挿入（AI要約対象） |
| ④ | 対応状況で入力ありの場合 | → 患者対応・外部対応それぞれの内容を挿入（AI要約対象） |
| ⑤ | 大分類「Ⅲ.管理薬剤の紛失」を選択 | → 患者様の状態・反応の質問をスキップ |
| ⑥ | AI要約対象の自由入力欄に15文字以上入力 | → 要約確認フェーズを挿入 |
| ⑦ | 速報判断Ⅴ-1「服用あり」を選択 | → Ⅴ-2「健康被害の有無」を挿入 |
| ⑧ | 速報判断Ⅴ-2「健康被害あり」を選択 | → Ⅴ-3「入院の有無」を挿入 |
| ⑨ | 速報判断Ⅰで「施設在宅あり」を選択 | → 施設名テキスト入力を挿入 |

### 4-2. 動的分岐の実装方式

動的分岐は「回答を受け取った直後」に挿入する方式で実装している。これにより、現在の質問が最後の質問であっても、回答に応じた追加質問が確実に挿入される。

```javascript
function handleAnswer(answer) {
  // 1. バリデーションチェック
  // 2. 回答を保存
  // 3. 動的分岐：次の質問の前に追加質問を挿入
  if (q.key === 'taken' && answer === '服用した') {
    qs.splice(insertAt, 0, takenCountQ);
  }
  if (q.key === 'health' && answer.startsWith('あり')) {
    qs.splice(insertAt, 0, healthDetailQ);
  }
  if (q.key === 'current_action' && answer !== '未対応') {
    qs.splice(insertAt, 0, actionDetailQ);
  }
  // 4. 次の質問へ
}
```

## 第5章　AI要約機能

### 5-1. 機能の目的

自由入力された長文を、AIが簡潔に整理して確認を経てから記録する。文章力の個人差を吸収し、Slack通知と管理表の可読性を一律に保つ。店舗スタッフは要点を整理して書く時間もスキルも十分でないという制約を踏まえた機能。

### 5-2. AI要約の対象項目（8項目）

| No. | 項目 | 発火条件 |
|:---:|---|---|
| 1 | **疑義照会漏れ内容** | 調剤ミス＞疑義照会漏れの場合のみ |
| 2 | **症状詳細** | 健康被害「あり」を選択した場合のみ |
| 3 | **個人情報漏洩詳細** | 個人情報漏洩カテゴリの内容欄 |
| 4 | **クレーム内容** | クレームカテゴリの内容欄 |
| 5 | **事故概要** | 事故カテゴリの内容欄 |
| 6 | **法令違反内容** | 法令遵守違反カテゴリの内容欄 |
| 7 | **その他状況** | その他カテゴリの状況欄 |
| 8 | **対応内容** | 対応状況「未対応」以外を選択した場合のみ |

### 5-3. 要約確認のフロー

ユーザーが自由入力欄にテキストを入力すると、以下のフローで要約確認が実施される。

1. **Step 1：** 店舗スタッフが自由入力欄に回答を入力
2. **Step 2：** AIが要約処理を実行（言いよどみ・冗長表現を整理）
3. **Step 3：** 要約結果を提示「この内容で良いですか？」
4. **Step 4a：** 「はい」を選択 → 要約版を最終データとして保存し次の質問へ
5. **Step 4b：** 「いいえ」を選択 → 元の入力に戻して再入力を促す

### 5-4. 要約スキップ条件

> 以下の場合は要約確認をスキップ：
> - 入力が **15文字未満** の場合（要約しても元の入力と大差ないため）
> - 要約処理の結果が元の入力と **完全に同一** の場合
>
> スキップした場合はそのまま次の質問に進む。

### 5-5. プロトタイプの要約処理

プロトタイプ実装ではルールベースの簡易要約を採用している。以下のような処理を実施する。

- **言いよどみの削除：** 「えーっと」「あのー」「えっと」「あの、」「その、」など
- **冗長表現の削除：** 「という感じで」「みたいな」「と思います」「なんというか」など
- **文末の整形：** 「ですね、」「んですが」などの整理、句読点の正規化
- **緩衝語の削除：** 「ちょっと」「少し」など
- **最終整形：** 末尾の余分な空白・読点を削除し、適切な句点を付加

### 5-6. 実運用版での要約処理

プロトタイプではルールベース処理だが、実運用ではClaude API等のLLM呼び出しに置き換える。LLM呼び出し時に担保すべき品質要件は以下の通り。

- 文意を変えず、事実情報を保持すること
- 5W1Hを意識した整理（いつ・どこで・誰が・何を・なぜ・どうした）
- Slack通知で一読できる長さに圧縮
- 専門用語・薬剤名は原文のまま保持（AIによる勝手な変更を防ぐ）

## 第6章　修正機能（編集フロー）

### 6-1. 修正機能の3パターン

AIエージェントは、対話の任意のタイミングで入力済み項目を修正できる。誤入力への心理的ハードルを下げ、最初からやり直す必要をなくす。

| 修正パターン | 動作 |
|---|---|
| **会話途中の修正** | 質問への回答中でも、入力欄の上部にある「← 前の回答を修正する」リンクから入力済みの任意項目を選んで修正可能。修正後は元の質問に戻って続行 |
| **確認画面での修正** | 最終確認画面で「修正する」を押すと、全入力項目がボタン一覧で表示。修正したい項目だけを選んで値を変更。変更後は確認画面に戻る。最初からやり直す必要はない |
| **分類変更時の連動** | 「分類」や「ミス種類」を変更した場合、関連する追加質問が変わるため、その部分だけやり直し。共通項目（店舗名等）は保持される |

### 6-2. 修正時のAI要約再実行

AI要約対象の項目（症状詳細・対応内容・事故概要等）を修正画面から編集した場合、再度AI要約が実行される。要約結果に対して「要約を採用」「元の文のまま」を選択できる。

## 第7章　バリデーション

### 7-1. 店番のバリデーション

店番の入力は、**6桁の数字**であることを必須とする。入力された値がこの条件を満たさない場合、エラーメッセージを表示し、再入力を促す。

```javascript
validate: (v) => /^\d{6}$/.test(v.trim()) ? null : '店番は6桁の数字で入力してください'
```

バリデーションは以下のタイミングで動作する。

- **初回入力時：** 質問への回答時に検証
- **修正画面での編集時：** 修正値の入力時に検証

### 7-2. その他の入力ルール

- 店番と店舗名は**両方の入力が必須**（どちらか一方では不可）
- 補足事項は**任意入力**。「なし」と入力するか空欄でも次に進める
- 自由入力欄全般：プレースホルダーで具体例を提示し、入力を誘導する

## 第8章　出力仕様（Slack通知・管理表）

### 8-1. Slack通知のフォーマット

確認画面で「確定」を押すと、すべての報告がSlackチャンネル **`#ホットライン報告`** に自動投稿される。AIエージェントによる判定は行わず、すべての報告を一律に通知する設計である。

通知内容には以下の項目が含まれる。

- 分類（大分類・サブ分類）

- 投薬日／発覚日

- 正誤情報
- 服用の有無・回数（調剤ミスのみ）

- 対応状況・対応内容
- 補足事項

- `@PSV`（該当地域の担当）

## 第9章　実装方針と今後の拡張

### 9-1. プロトタイプの実装範囲

現在のプロトタイプ（`hotline_ai_agent_v3.jsx`）は、UI／UX検証を目的としたReact単体での実装である。Slack通知や管理表への自動記入は画面上のモック動作（「通知しました」「記入しました」と表示するのみ）となっている。

プロトタイプで検証済みの要素は以下の通り。

- 対話フロー全体の妥当性
- 質問順序の自然さ（記憶の流れに沿うか）
- 動的分岐の動作
- AI要約の有効性（ルールベース版）
- 修正機能の操作性
- バリデーションの動作

### 9-2. 実運用版で必要な追加実装

プロトタイプから実運用版に移行するためには、以下の機能拡張が必要となる。

| 機能 | 概要 |
|---|---|
| **Slack API連携** | 実際のSlackチャンネルへの自動投稿、メンション先の動的取得（地域・店舗からPSVを特定） |
| **Google Sheets API連携** | 管理表への自動転記、24列フォーマットへのマッピング |
| **Claude API連携** | ルールベース要約からLLMベース要約への置き換え |
| **認証・認可** | 店舗スタッフのID連携、店番の自動補完 |
| **セッション管理** | 途中保存、複数端末からのアクセス、ログ保管 |
| **Slack返信→管理表自動更新** | PSVのSlack返信内容をAIが検知し、管理表のステータス・経過欄に自動反映 |
| **未完了案件の自動リマインド** | 一定期間ステータスが動かない案件の自動督促 |
| **ダッシュボード** | 未完了案件・傾向分析の可視化 |

### 9-3. 機能拡張の方向性

将来的な機能拡張として、以下が考えられる。

- **調剤システム連携：** 処方変更・返品入力時に「ホットライン報告が必要かもしれません」とアラート
- **過去事例の参照機能：** 類似事例の検索、対応履歴のサジェスト
- **自動対応提案：** 軽微案件についてAIが対応方法の参考案を提示（最終判断は人間）
- **音声入力対応：** チャット入力に加え、音声でも報告可能に
- **多言語対応：** 将来的な海外展開に備えた言語切替

### 9-4. プロトタイプから検証フェーズへ

本プロトタイプは、企画書の**対策案A（AIエージェント方式）**に該当する。今後、対策比較（Step 02）の中でPoC対象として実装範囲・必要工数の精緻化を行い、選定された場合は詳細運用設計（Step 03）でシステム要件・移行計画を策定する。
