# ホットラインAIエージェント 機能要件プロンプト 更新版

> **対象プロジェクト：** `sugi_hotline_system`  
> **対象ファイル：** `index.html` / `style.css` / `app.js`  
> **コンセプト：** 店舗インシデントの対話型ヒアリング × 動的分岐 × AI要約 × 音声入力 × Slack/管理表連携  
> **UI方針：** スギアプリに近い安心感のあるグリーン基調、スマホファースト、PCでは左右2カラム  
> **更新日：** 2026年5月13日  
> **ステータス：** ブラウザ単体プロトタイプ。Slack送信、Google Sheets転記、LLM要約APIはモック。

---

## 0. このプロンプトの目的

この文書は、店舗スタッフがブラウザ上でホットラインへインシデントを相談するためのプロトタイプを、同一UI基盤のまま保守・拡張するための機能要件プロンプトである。

今後、AIまたは開発者が本プロジェクトを改修する場合は、以下を厳守する。

1. **現行UI基盤を崩さない。**
   - `index.html` / `style.css` / `app.js` の静的構成を維持する。
   - スマホ画面では、現在の縦型UIを維持する。
   - PC画面では、左にチャットウインドウ、右に入力ウインドウを配置する。
2. **店舗スタッフが忙しい状態でも迷わないUIにする。**
   - 選択式を優先する。
   - 入力文の整形はAI要約側で吸収する。
   - 不明な情報は「不明」または自由入力で進められるようにする。
3. **AIは重大度・ミスレベルを判定しない。**
   - AIは聞き取り、要約、整形、通知文作成を支援する。
   - 専門判断、ミスレベル判定、対応判断はホットライン担当者または関係部署が行う。
4. **すべての報告をSlackへ通知する。**
   - 速報対象か通常報告かは表示するが、報告の送信可否はAIが分岐しない。
5. **音声入力は補助機能として扱う。**
   - 画面上は控えめなマイクボタンのみを配置する。
   - 音声が使えないブラウザでも、ボタン選択・テキスト入力で全フローを完了できること。

## 1. システム概要

### 1-1. 目的

店舗で発生した以下のようなトラブルについて、店舗スタッフがチャット形式でAIエージェントに回答するだけで、必要情報を構造化して本部・ホットラインへ共有できるようにする。

- 調剤ミス
- 個人情報漏洩
- 管理薬剤の紛失
- クレーム、事故、法令遵守違反、その他トラブル

### 1-2. 想定利用者

- 店舗薬剤師
- 店舗スタッフ
- 店長、管理薬剤師
- ホットライン受付担当
- PSV、営業部長、統括部長、リスク担当

### 1-3. 利用端末

- 主端末：スマートフォン
- 副端末：PC、タブレット

### 1-4. プロトタイプの範囲

現在のプロトタイプはブラウザ単体で動作する。

実装済みまたはモック化済みの範囲は以下。

| 領域 | 状態 |
|---|---|
| 対話型ヒアリング | 実装済み |
| 動的分岐 | 実装済み |
| ルールベースAI要約 | 実装済み |
| 項目単位修正 | 実装済み |
| 音声入力・音声選択 | 実装済み。ブラウザ依存 |
| Slack投稿プレビュー | 実装済み |

| Google Sheets実転記 | モック |

| 認証・権限 | 未実装。将来拡張 |

## 2. UI/UX要件

### 2-1. 基本デザイン

- スギアプリに近いグリーン基調の配色とする。
- 背景は淡いグリーンまたは明るいニュートラル色とする。
- 主要ボタンは視認性の高いグリーン系とする。
- カード、ボタン、入力欄は角丸を強めにする。
- 文字は読みやすさを優先し、以下を基本とする。
  - `Hiragino Sans`
  - `Yu Gothic`
  - `Meiryo`
  - system font
- 高齢者や慌てている店舗スタッフでも読めるよう、スマホでは本文16px以上、選択ボタン17px前後を基本とする。
- 質問タイトルは太く大きく表示する。
- 補足文は落ち着いたグレーで表示する。

### 2-2. スマホレイアウト

スマホでは、現在のUIを維持する。

```text
┌──────────────────────┐
│ ヘッダー              │
│ タイトル / 文字拡大 / 最初から │
├──────────────────────┤
│ チャットエリア        │
│ AIとユーザーの会話履歴 │
├──────────────────────┤
│ 入力エリア            │
│ 質問タイトル / 選択肢 / 入力欄 │
├──────────────────────┤
│ フッター              │
│ 回答を修正 / 電話で相談 │
└──────────────────────┘
```

#### スマホ固有要件

- `viewport-fit=cover` を指定する。
- safe-area を考慮し、ノッチやホームインジケータで操作ボタンが隠れないようにする。
- ヘッダー・フッターを固定的な領域として設ける。
- チャットエリアと入力エリアは、画面内でそれぞれ縦スクロールできる。
- 横スクロールは禁止する。
- 画面拡大、ピンチズーム、ダブルタップ拡大、Ctrl+ホイール拡大は抑止する。
- キーボード表示時に入力欄が見切れにくいよう、`visualViewport` の高さをCSS変数に反映する。

### 2-3. PCレイアウト

画面幅960px以上では、左右2カラムへ切り替える。

```text
┌──────────────────────────────────────────┐
│ ヘッダー                                  │
├──────────────────────┬───────────────────┤
│ 左：チャットウインドウ │ 右：入力ウインドウ │
│ 会話履歴              │ 質問・選択・入力    │
│ 独立スクロール        │ 独立スクロール      │
├──────────────────────┴───────────────────┤
│ フッター                                  │
└──────────────────────────────────────────┘
```

#### PC固有要件

- 左側にチャットウインドウ、右側に入力ウインドウを表示する。
- 左右のカードは同じ高さに揃える。
- 右側入力エリアが長くなっても、画面全体ではなく入力ウインドウ内で縦スクロールする。
- 1200px以上ではチャット側をやや広めにして、Slackプレビューや長文会話を読みやすくする。
- スマホUIに影響しないよう、PC切替はCSSメディアクエリで行う。

### 2-4. 音声入力UI

- 音声入力は、各質問のタイトル右側に小さな丸いマイクボタンのみ表示する。
- ボタン文言は常時表示しない。
- `aria-label` と `title` には「音声で選択する」または「音声で入力する」を設定する。
- 音声認識中は、マイクボタンに聞き取り中状態を表示する。
- 聞き取り結果やエラーは必要時のみ小さく表示する。
- 音声認識非対応ブラウザでは、マイクボタンを無効化し、通常入力を促す。

## 3. 画面構成

### 3-1. `index.html`

必須構造は以下。

```text
app-shell
├─ app-header
│  ├─ タイトル
│  ├─ 文字を大きく
│  └─ 最初から
├─ app-main
│  └─ chat-window
│     ├─ bot-area
│     │  └─ chatLog
│     └─ answer-panel
│        ├─ progressText
│        └─ questionArea
└─ app-footer
   ├─ 回答を修正
   └─ 電話で相談
```

### 3-2. `style.css`

- UIの見た目、スマホ/PCレスポンシブ、横スクロール抑止、safe-area、Slackプレビュー表示を担当する。
- インラインCSSは使用しない。
- PC切替は `@media (min-width: 960px)` を基準にする。

### 3-3. `app.js`

- 質問定義、状態管理、分岐制御、要約、音声入力、Slackプレビュー、確認・修正・完了画面を担当する。
- インラインイベントは使用しない。
- 画面描画後に `addEventListener` でイベントを紐づける。

## 5. 共通ヒアリング項目

大分類選択後、全カテゴリ共通で以下を確認する。

|---|---|---|---|---|

店番入力後、システムは店舗マスタから以下を自動反映する。

| 自動反映項目 | 説明 |
|---|---|
| 営業部 | 店舗が所属する営業部 |
| 地区名 | 店舗が所属する地区 |
| PSV名 | 該当店舗の担当PSV |

プロトタイプではモックマスタまたは店番末尾からの仮算出とする。実運用では店舗マスタAPIまたは管理テーブルから取得する。

### 11-1. 分岐一覧

| No. | トリガー | 動作 |
|---:|---|---|
| 1 | 大分類が調剤ミス | ミス種類14分類を挿入 |
| 2 | ミス種類が渡し忘れ | 正誤情報の質問文を渡し忘れ専用に変更 |
| 3 | ミス種類が異薬異物混入 | 正誤情報の質問文を本来薬剤/混入物専用に変更 |
| 4 | ミス種類が別患者への投薬 | 正誤情報の質問文を本来患者/実際患者専用に変更 |
| 5 | ミス種類が疑義照会漏れ | 正誤情報をAI要約対象にする |
| 6 | 服用の有無が服用した | 服用回数を追加 |
| 7 | 健康被害があり | 症状詳細を追加しAI要約対象にする |
| 8 | 大分類が管理薬剤の紛失 | 患者様・関係者の状態をスキップ |
| 9 | 施設在宅関与があり | 施設名を追加 |
| 10 | 調剤ミス以外で速報判断の服用あり | 速報判断の健康被害を追加 |
| 11 | 健康被害あり | 入院の有無を追加 |
| 12 | AI要約対象の自由入力が15文字以上 | 要約確認フェーズを挿入 |
| 13 | 分類を修正 | 分類依存項目をクリアし、共通項目は保持 |
| 14 | ミス種類を修正 | 正誤情報をクリアし、再入力させる |

### 11-2. 分岐実装方式

- 現在の全質問リストを `makeQuestions()` で都度再生成する。
- 各質問に `condition` を持たせ、条件に合う項目だけ表示する。
- 回答保存後に不要になった回答を `cleanupAfterAnswer()` で削除する。
- これにより、分類変更や回答変更に連動して質問群を再構成できる。

## 12. AI要約要件

### 12-1. 対象

以下の項目はAI要約対象。

- 疑義照会漏れ時の正誤情報
- 症状詳細
- 発覚の経緯
- 対応状況（患者）
- 対応状況（外部）
- 個人情報漏洩の「誰の何を誰に渡したか」
- 回収対応
- 漏洩先への対応
- 捜索状況
- 事件・事故の概要
- 事件・事故の内容詳細
- 発生後の対応
- 健康被害の内容

### 12-2. 要約発火条件

- 対象項目であること。
- 入力が15文字以上であること。
- 要約結果が元文と完全一致しないこと。

### 12-3. 要約確認UI

要約案を提示し、以下から選択できるようにする。

1. この要約を採用する
2. 元の文章のまま使う
3. もう一度書き直す

### 12-4. 要約品質要件

実運用でLLMへ置き換える場合は以下を守る。

- 文意を変えない。
- 薬剤名、数量、規格、患者状態、日付、機関名を勝手に変更しない。
- 不明な内容を補完しない。
- 断定できない内容は断定しない。
- Slackで読みやすい長さに整理する。
- 5W1Hを意識して時系列を整える。

## 13. 音声入力要件

### 13-1. 技術方式

- ブラウザの `SpeechRecognition` または `webkitSpeechRecognition` を使用する。
- 言語は `ja-JP` とする。
- `interimResults = true` とし、途中結果も表示できるようにする。
- `continuous = false` とし、1回のマイク押下で1回答を基本とする。
- 非対応ブラウザでは通常入力へフォールバックする。

### 13-2. 選択式の音声入力

選択式では、話した内容を選択肢にマッチングする。

例：

| 発話 | 選択される値 |
|---|---|
| 調剤 | Ⅰ. 調剤ミス |
| 個人情報 | Ⅱ. 個人情報漏洩 |
| 紛失 | Ⅲ. 管理薬剤の紛失 |
| 事故 | Ⅳ. その他トラブル |
| あります | あり |
| ありません | なし |
| 飲みました | 服用した |
| 飲んでいません | 服用していない |
| 分からない | 不明 |

### 13-3. テキスト式の音声入力

- 音声認識結果を入力欄に反映する。
- 自動送信はしない。
- ユーザーが内容を確認してから「入力した内容で次へ進む」を押す。
- 既存入力がある場合は追記する。
- テキストエリアの場合は改行区切り、1行入力の場合はスペース区切りを基本とする。

## 14. 修正機能要件

### 14-1. 会話途中の修正

- フッターの「回答を修正」から、入力済み項目を一覧表示する。
- 修正したい項目だけを選んで編集できる。
- 修正後は元の質問または確認画面へ戻る。

### 14-2. 確認画面での修正

- 送信前の確認画面では、各項目に「修正する」ボタンを表示する。
- 1項目だけ修正して確認画面へ戻れる。
- 最初からやり直す必要はない。

### 14-3. 分類変更時の連動削除

- 大分類変更時は、分類依存項目を削除する。
- 本部区分、店番、店舗名などの共通項目は可能な限り保持する。
- ミス種類変更時は、正誤情報を削除し再入力させる。
- 条件分岐で不要になった項目は削除する。

## 16. Slack送信プレビュー要件

### 16-1. 基本方針

- 最終確認画面と受付完了画面にSlack投稿プレビューを表示する。
- 項目の羅列ではなく、ブロック単位で整形する。
- 速報対象の場合は目立つバナーを表示する。
- 通常報告の場合も、速報条件への該当なしを明示する。
- 完了画面では、Slackへ貼り付けやすいテキスト形式も表示する。

### 16-2. 表示ブロック

Slackプレビューは以下の順で表示する。

1. 受付情報
2. 速報判断
3. 店舗情報

5. 事案内容
6. 対応状況
7. 補足

## 22. 改修時の禁止事項

- スマホUIを崩す変更。
- PCレイアウトのためにスマホで横スクロールを発生させる変更。
- HTMLへインライン `style`、`onclick`、`onchange` を追加する変更。
- AIがミスレベル、重大度、エスカレーション要否を最終判断する仕様。
- 音声入力が使えない場合にフロー完了できなくなる仕様。
- Slackプレビューを単なる長文羅列に戻す変更。
- 店舗スタッフに専門判断を求める質問追加。
- 分類変更時にすべての入力を消す変更。

## 23. 受け入れ条件

### 23-1. UI

- スマホではヘッダー、チャット、入力、フッターの縦構成を維持している。
- PC幅960px以上では、左チャット、右入力の2カラムになる。
- 横スクロールが発生しない。
- 縦スクロールは可能。
- 画面拡大抑止が効いている。
- マイクボタンは控えめな丸ボタンのみである。

### 23-2. フロー

- 4カテゴリすべてで確認画面まで到達できる。
- 調剤ミスでは14分類が表示される。
- 服用した場合のみ服用回数が出る。
- 健康被害ありの場合のみ症状詳細が出る。
- 管理薬剤紛失では患者状態が出ない。
- その他トラブルは速報対象になる。
- 速報理由がSlackプレビューに表示される。

### 23-3. 入力補助

- 15文字以上の要約対象項目で要約確認が表示される。
- 要約採用、元文採用、書き直しが選択できる。
- 音声で選択肢を選べる。
- 音声でテキスト欄へ入力できる。
- 音声非対応でも通常操作で完了できる。

### 23-4. 出力

- Slackプレビューがブロック単位で整形される。
- 完了画面でSlackテキスト形式を確認できる。
- メンションに担当PSV、営業部長、統括部長が含まれる。
- 速報対象ではリスク担当が追加される。
